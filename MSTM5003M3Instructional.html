PKI Fundamentals

Understanding the foundation of Public Key Infrastructure

What is PKI?

A Public Key Infrastructure (PKI) provides all the components necessary for different types of users and entities to communicate securely and in a predictable manner. It consists of hardware, applications, policies, services, programming interfaces, cryptographic algorithms, protocols, users, and utilities.

Key PKI Components

Certificate Authority (CA) Trusted authority that certifies identities
Registration Authority (RA) Accepts certificate requests and authenticates users
Certificate Revocation List (CRL) List of revoked certificate serial numbers
Certificate Repository Centralized directory for certificates and public keys

Digital Certificates

Understanding X.509 certificates and their components

Certificate Structure (X.509 v3)

Digital certificates bind an individual's identity to a public key using the X.509 standard format. They contain all the information a receiver needs to be assured of the public key owner's identity.

Certificate Classes

Certificate Extensions

Key Usage Extensions: Dictate how the public key can be used (encryption, digital signatures, etc.)

Critical vs Non-Critical: Critical extensions must be understood by the receiver, non-critical can be ignored

Certificates move through these stages to ensure security and trustworthiness over time.

Key Points

  • Registration: User or device identity verified by a Registration Authority (RA) before a certificate is issued.
  • Generation: Certificate Authority (CA) creates the certificate and digitally signs it with its private key.
  • Distribution & Usage: Certificates are distributed and used for encryption, authentication, and digital signatures.
  • Revocation / Expiration: Certificates can expire or be revoked (via CRL or OCSP) if compromised or no longer needed.

Security Convergence

Integrating physical and information security

Convergence Benefits

Convergence combines physical and information security to improve identification of unauthorized activity. For example:

Scenario: Anomaly Detection

If an access control system approves access from an outside IP address, but the physical security system shows the user is in the building, an anomaly exists and should be investigated.

Protection Against Cloned Credentials

Convergence significantly improves defenses against cloned credentials by correlating physical presence with logical access attempts.

TEMPEST and Electronic Emanations

TEMPEST: Technology to prevent disclosure through electronic emanations

Van Eck Emissions: Electromagnetic radiation that can be intercepted to reconstruct displayed information

Countermeasures: Faraday cages, proper cable shielding, and controlled electromagnetic environments

Security Scenarios

Apply your knowledge to real-world situations

Scenario 1: Certificate Compromise

Situation: An employee reports their laptop was stolen from their car. The laptop contained their digital certificate and private key.

Question: What immediate actions should be taken?

Solution:
  1. Immediately revoke the certificate (add to CRL)
  2. Notify the Certificate Authority
  3. Generate new key pair and request new certificate
  4. Update OCSP status if available
  5. Investigate potential data exposure

Scenario 2: Physical Security Breach

Situation: Security cameras show an unauthorized person in the server room at 2 AM. They were seen inserting a USB device into a server.

Question: What security controls failed and what should be the response?

Failed Controls & Response:
  • Failed: Layered access, after-hours monitoring, USB port security
  • Response: Immediate server isolation, forensic imaging, malware scanning, access log review, strengthen USB policies

Scenario 3: Trust Model Selection

Situation: A company with 5 subsidiaries needs to implement PKI. Each subsidiary wants autonomy but they need to share certificates for B2B transactions.

Question: Which trust model would be most appropriate?

Recommended: Hybrid trust model with a bridge CA. Each subsidiary maintains its own hierarchical CA structure while the bridge CA handles cross-certification between subsidiaries, providing scalability and autonomy.
